## Decisions Made

- **postgres.js over alternative drivers:** Drizzle's documented default for Supabase Postgres; all the RESEARCH.md code examples use it; Neon serverless driver would only help if we deployed to Cloudflare Workers, which we aren't.
- **`drizzle-kit generate+migrate` over `push`:** Plan 04 (auth) and Phase 3 (loyalty) need a reproducible migration trail. `push` skips history; `generate+migrate` commits SQL files. Chosen deliberately so `npx drizzle-kit migrate` on a fresh clone reproduces the exact schema.
- **CHECK constraints over Postgres enum types:** When Phase 3 decides reward economics and needs to add a new `point_transaction.type` value (e.g., `expiry_burn`), the migration is one line in schema.ts plus a generated ALTER TABLE DROP/ADD CONSTRAINT. With Postgres enums it would require an `ALTER TYPE ... ADD VALUE ...` which cannot run in a transaction. CHECK constraints trade a tiny amount of type-safety for a large amount of migration flexibility.
- **Single schema.ts file:** 9 tables at ~110 lines fits comfortably in one file and matches Drizzle's own documentation examples. If Phase 3 pushes the table count past ~20, a barrel-import pattern (`schema/user.ts`, `schema/loyalty.ts`, re-exported from `schema/index.ts`) is trivial to introduce later.
- **Extracted ANON_KEY via JSON output:** CLI 2.89.1's pretty-print `supabase status` only shows the new-style `sb_publishable_*` key, not the legacy JWT `eyJ...` anon key. The JSON output (`-o json`) still emits both `ANON_KEY` (legacy JWT) and `PUBLISHABLE_KEY` (new-style). The plan acceptance criteria literally require `PUBLIC_SUPABASE_ANON_KEY=eyJ`, and all the @supabase/supabase-js + @supabase/ssr documentation still uses JWT anon keys, so I extracted from JSON and wrote the JWT into .env. This is a friction point worth flagging in the README for anyone onboarding.
- **supabase as a devDep, not global:** The plan allowed fall-back to global install if npm package failed. It didn't fail (installed clean at 2.89.1), so the npm path was taken -- team members now only need Node 22 + Docker Desktop, no extra tooling.

**2. [Rule 3 - Blocking] CLI 2.89.1 hides legacy JWT anon key in pretty-print output**
- **Found during:** Task 2 (step 4: capture anon key from `supabase status`)
- **Issue:** Plan acceptance criteria require `PUBLIC_SUPABASE_ANON_KEY=eyJ` (JWT prefix) and the plan's step-4 wording says "Extract the `anon key` line (it looks like `anon key: eyJ...`)". In CLI 2.89.1 the pretty-print `supabase status` output no longer shows an `anon key:` line at all -- it shows `Publishable: sb_publishable_*` and `Secret: sb_secret_*` (the new-style Supabase API keys). The legacy JWT anon key is only accessible via `supabase status -o json` which still emits it under the `ANON_KEY` field.
- **Fix:** Extracted via `npx supabase status -o json | node -e "...parse ANON_KEY..."` and wrote the JWT into `.env`. The key present in the running local stack is still a valid JWT signed with the canonical local JWT_SECRET, so @supabase/supabase-js and @supabase/ssr will both accept it in later plans.
- **Files modified:** `.env` (gitignored, not committed)
- **Verification:** `grep -c "PUBLIC_SUPABASE_ANON_KEY=eyJ" .env` returns 1. Running the query verifies it starts with `eyJhbGciOi...` which is the expected JWT header prefix.
- **Committed in:** N/A (.env is gitignored)
